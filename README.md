# On-Policy CoT Distillation with Lightweight Rationale Compression

This project implements **logits-level on-policy Chain-of-Thought distillation** for mathematical reasoning.  
The teacher model is `Qwen2.5-7B-Instruct`, and the student model is `Qwen2.5-1.5B-Instruct`.

The goal is to train the student not only to produce correct final answers, but also to generate short, useful reasoning traces. The key idea is to combine:

1. **off-policy cold start**, where the teacher generates long rationales and compresses them into difficulty-aware concise rationales for student SFT;
2. **on-policy distillation**, where the student generates its own reasoning trajectories, and the teacher provides logits-level supervision on those trajectories;
3. **lightweight CoT compression**, where reasoning supervision is kept concise rather than verbose.

The main task is mathematical word-problem reasoning, with GSM8K as the default dataset.


## 1. Overall Logic

The full training pipeline has two stages.

```text
Stage A: Difficulty-aware compressed CoT cold start

question
→ teacher generates long CoT + answer
→ valid long CoT is compressed with difficulty-aware instructions
→ student is trained on compressed teacher targets
→ obtain cold-start student checkpoint

Stage B: On-policy logits-level CoT distillation

question
→ cold-start student generates rationale + answer
→ teacher evaluates the same student trajectory
→ teacher logits are computed along the student-generated sequence
→ student is trained with logits-level distillation on its own trajectories
```

- The cold-start stage is included because directly applying on-policy distillation to a weak base student is unstable. The student may initially produce malformed reasoning, incorrect answer formats, or very noisy trajectories. Off-policy cold start brings the student into a usable region before on-policy training begins.
- The actual main method is still on-policy because the distillation sequences in Stage B are generated by the current student model.

## 2. Models
Teacher: Qwen/Qwen2.5-7B-Instruct
1. generating compressed rationales for cold start;
2. producing logits-level supervision during on-policy distillation;
3. optionally correcting or compressing student-generated rationales for analysis.
Student: Qwen/Qwen2.5-1.5B-Instruct


## 3. Dataset

Use GSM8K as the default dataset.

Each example should be converted into the following internal format:

```json
{
  "id": "gsm8k_train_000001",
  "question": "...",
  "gold_answer": "..."
}
```

The original GSM8K answer field contains both the human-written rationale and the final answer.
For this project, the human-written rationale is not used as the main compression source. It is only used to extract the gold final answer.

The final answer should be extracted from the part after `####`.

Example:
```text
Raw answer:
Janet has 3 apples and buys 5 more. So she has 3 + 5 = 8.
#### 8

Extracted gold_answer:
8
```

- The gold_answer field is used for filtering teacher generations, filtering compressed rationales, and final evaluation.

## 4. Output format

```text
### Rationale
{compressed reasoning}

### Answer
{final answer}
```

- The rationale should be concise. It should contain only the key reasoning steps needed to reach the final answer.

## 5. Stage A: Difficulty-Aware Cold Start Data

Stage A constructs the cold-start distillation data. Instead of directly asking the teacher for a short rationale, it first generates a detailed rationale and then compresses it.

The full process is:

```text
question
→ teacher generates long CoT + answer
→ keep examples whose teacher answer matches gold_answer
→ estimate difficulty from long-CoT length
→ compress the rationale with difficulty-aware instructions
→ keep valid compressed examples
→ train student on compressed rationale + answer
```

The resulting compressed rationale is used as the cold-start distillation target for the student.

### 5.1 Teacher Long CoT Generation

For each training question, use the teacher model to generate a detailed rationale and final answer.

Teacher long-CoT prompt:
```text
You are a math reasoning teacher.

Given a math word problem, solve it step by step and provide the final answer.

Requirements:
1. Show the complete reasoning process.
2. Include all necessary intermediate computations.
3. The final answer should be short and easy to parse.
4. Use exactly the following format:

### Long Rationale
...

### Answer
...

Question:
{question}
```

Save the generated output with these core fields:
```json
{
  "id": "...",
  "question": "...",
  "gold_answer": "...",
  "teacher_long_response": "### Long Rationale\n...\n\n### Answer\n...",
  "teacher_long_rationale": "...",
  "teacher_long_answer": "..."
}
```

Filter the long-CoT generations before compression.

Keep an example only if:
1. the teacher response contains both ### Long Rationale and ### Answer;
2. the long rationale is non-empty;
3. the teacher answer is non-empty and parseable;
4. the parsed teacher answer matches the gold answer.

Discard teacher generations whose final answer does not match the gold answer.

### 5.2 Difficulty Estimation

After obtaining valid teacher-generated long CoTs, estimate difficulty from the long rationale itself.

The default difficulty score is:
difficulty_score = token_count(teacher_long_rationale)

Then split the valid training examples into three buckets by quantiles:
easy:   bottom 33%
medium: middle 33%
hard:   top 33%

Store the difficulty information in each example:
```json
{
  "id": "...",
  "question": "...",
  "gold_answer": "...",
  "teacher_long_rationale": "...",
  "teacher_long_answer": "...",
  "difficulty_score": 96,
  "difficulty_level": "medium"
}
```

This difficulty label is only used to control the compression strength during Stage A.
No additional hard dataset is introduced.
It is a lightweight proxy for difficulty, not a ground-truth difficulty label.

### 5.3 Difficulty-Aware Compression Policy

The compression strength depends on the difficulty bucket.

```text
easy:
  Use one short reasoning sentence.
  Keep only the essential operation.

medium:
  Use two concise reasoning sentences.
  Preserve the main intermediate relation and final computation.

hard:
  Use several concise reasoning steps.
  Preserve all necessary intermediate computations, but remove repetition and self-reflection.
```

The goal is not to make every rationale equally short.
The goal is to remove redundancy while preserving enough reasoning structure for the student to learn.

### 5.4 Teacher Compression Prompt

Use the teacher model to compress its own long rationale.

```text
You are a rationale compression module for math reasoning distillation.

Given a math problem, a teacher-generated long rationale, the gold final answer,
and a difficulty level, compress the rationale according to the difficulty level.

Compression policy:
- Easy: keep only one essential reasoning sentence.
- Medium: keep two concise reasoning sentences.
- Hard: keep all necessary intermediate computations, but remove repetition and self-reflection.

Requirements:
1. Preserve the reasoning needed to derive the final answer.
2. Remove redundant restatement, self-reflection, and unnecessary explanation.
3. Do not change the final answer.
4. The final answer must match the gold final answer.
5. Use exactly the following format:

### Rationale
...

### Answer
...

Question:
{question}

Teacher-generated long rationale:
{teacher_long_rationale}

Gold final answer:
{gold_answer}

Difficulty level:
{difficulty_level}
```

Save the compressed output as:
```json
{
  "id": "...",
  "question": "...",
  "gold_answer": "...",
  "teacher_long_rationale": "...",
  "teacher_long_answer": "...",
  "difficulty_score": 96,
  "difficulty_level": "medium",
  "teacher_compressed_response": "### Rationale\n...\n\n### Answer\n...",
  "teacher_compressed_rationale": "...",
  "teacher_compressed_answer": "..."
}
```

### 5.5 Compression Statistics

During Stage A, record the token length of both the teacher-generated long rationale and the compressed rationale.

For each example, save:
```json
{
  "id": "...",
  "question": "...",
  "gold_answer": "...",
  "difficulty_score": 96,
  "difficulty_level": "medium",
  "long_rationale_tokens": 118,
  "compressed_rationale_tokens": 39,
  "compression_ratio": 0.3305,
  "token_reduction": 0.6695
}
```

Compute:
compression_ratio = compressed_rationale_tokens / long_rationale_tokens
token_reduction = 1 - compression_ratio

These statistics describe the cold-start data construction process.
They are not the main performance metric.

Optional: report average long length, compressed length, compression ratio, and token reduction overall and by difficulty bucket.
Expected behavior is that easy examples are compressed most aggressively, while hard examples preserve more intermediate computations.

### 5.6 Filtering Compressed Rationales

Filter teacher-compressed examples before cold-start training.

Keep an example only if:
1. the compressed response contains both ### Rationale and ### Answer;
2. the compressed rationale is non-empty;
3. the compressed answer is non-empty and parseable;
4. the parsed compressed answer matches the gold answer;
5. the compressed rationale is shorter than the teacher-generated long rationale;
6. optionally, the compressed rationale is not excessively short for medium/hard examples.

Condition 5 ensures that the compression module actually performs compression.
Condition 6 prevents the teacher from deleting important reasoning steps for harder examples.

The resulting dataset is the cold-start distillation dataset.

### 5.7 Student Cold-Start Training

Train the student on the filtered difficulty-aware compressed rationales.

Input:
```text
### Question
{question}
```

Target:
```text
### Rationale
{teacher_compressed_rationale}

### Answer
{teacher_compressed_answer}
```

Use response-only supervised training.
Prompt tokens should be masked with -100 in the labels, and loss should only be computed on the target response.

The output checkpoint from this stage is the cold-start student checkpoint.


## 6. Stage B: On-Policy Logits-Level CoT Distillation

The on-policy stage starts from the cold-start student checkpoint.

For each question, the current student generates its own response:
question → student rationale + student answer

This generated response is the on-policy trajectory.

Then the teacher is evaluated on the same trajectory, and the student is trained to match the teacher’s next-token distributions along that student-generated sequence.


## 7. On-Policy Rollout

For each training question, generate a student response using the current student checkpoint.

Save the rollout data as:
```json
{
  "id": "...",
  "question": "...",
  "gold_answer": "...",
  "student_response": "### Rationale\n...\n\n### Answer\n...",
  "student_rationale": "...",
  "student_answer": "...",
  "student_correct": true
}
```

- The student_correct field is computed by extracting the final answer and comparing it with the gold answer.
- This field is useful for analysis and optional filtering, but the on-policy logits-level distillation can use both correct and incorrect student trajectories.


## 8. Logits-Level Distillation Objective

For each on-policy rollout, construct the full sequence:
prompt + student_response

The prompt is:
```text
### Question
{question}
```

The response is the student-generated sequence:
```text
### Rationale
{student_rationale}

### Answer
{student_answer}
```

Both teacher and student are run on the same full sequence.

For causal language modeling, logits at position t predict the token at position t + 1.

Let:
z_T = teacher logits
z_S = student logits

Apply temperature scaling:
p_T = softmax(z_T / T)
p_S = log_softmax(z_S / T)

The token-level KL loss is:
KL(p_T || p_S)

Only response tokens should contribute to the loss. Prompt tokens should be masked out.

The logits-level distillation loss is:
L_KD = T^2 * mean_response_tokens KL(
    softmax(z_T / T) || log_softmax(z_S / T)
)

Optionally combine it with hard-label loss on the same student-generated trajectory:
L_CE = - log p_student(student_token | previous tokens)

The final loss is:
L = alpha * L_CE + beta * L_KD

where alpha and beta are configurable.

- The important point is that the sequence used for distillation is sampled from the student. Therefore, the teacher supervises the student on states that the student actually visits during generation.


## 9. Lightweight CoT Compression

The project uses compressed rationales instead of long CoT.

The cold-start data uses teacher-generated compressed rationales.
The on-policy data uses student-generated rationales, but the expected output format still encourages short rationales.

The rationale should usually contain:
1. the key quantities from the problem;
2. the necessary arithmetic or logical relation;
3. the final computation leading to the answer.

It should not contain:
1. long self-reflection;
2. repeated restatement of the question;
3. unnecessary checking phrases;
4. multiple alternative solution attempts.


## 10. Evaluation
Evaluate at least the following checkpoints:
1. cold-start student after Stage A;
2. on-policy student after Stage B.

The main metric is final-answer accuracy.

For each evaluation example:
1. generate student response;
2. extract the final answer from ### Answer;
3. normalize the extracted answer;
4. compare it with the normalized gold answer.

Save evaluation results as JSON:
```json
{
  "checkpoint": "...",
  "num_examples": 0,
  "accuracy": 0.0,
  "parse_fail_rate": 0.0,
  "avg_response_tokens": 0.0,
  "avg_rationale_tokens": 0.0
}
```

### 10.1 Difficulty-Bucket Evaluation

Optional: assign difficulty levels using teacher-generated long CoT length and report accuracy by bucket.

The procedure should match Stage A as closely as possible:
```text
eval question
→ teacher generates long CoT
→ keep valid long CoT if teacher answer matches gold answer
→ difficulty_score = token_count(teacher_long_rationale)
→ assign easy / medium / hard bucket
```

Use quantile thresholds computed from the training set unless a separate evaluation-specific analysis is desired.

Report accuracy both overall and by difficulty bucket:

```json
{
  "checkpoint": "...",
  "overall": {
    "accuracy": 0.0,
    "num_examples": 0
  },
  "easy": {
    "accuracy": 0.0,
    "num_examples": 0
  },
  "medium": {
    "accuracy": 0.0,
    "num_examples": 0
  },
  "hard": {
    "accuracy": 0.0,
    "num_examples": 0
  }
}
```

This analysis helps check whether the method improves easy, medium, or hard examples differently.
If teacher long-CoT generation fails for some evaluation examples, keep those examples in the overall evaluation, but exclude them from difficulty-bucket analysis.

### 10.2 Generated Length Evaluation

In addition to accuracy, record the generated response length:
avg_response_tokens
avg_rationale_tokens

These values are computed from the model-generated output, not from the cold-start teacher-compressed data.

This answers a different question from the compression statistics:
Compression statistics:
  How much shorter are the teacher-compressed cold-start rationales than the teacher-generated long rationales?

Generated length evaluation:
  How long are the rationales produced by the trained student at inference time?

### 10.3 Error Recovery Rate

When comparing the cold-start checkpoint and the on-policy checkpoint on the same evaluation set, compute error recovery rate:
error_recovery_rate =
number of examples wrong under cold-start but correct after on-policy
/
number of examples wrong under cold-start

Save it as:
```json
{
  "error_recovery_rate": 0.0,
  "wrong_before": 0,
  "wrong_before_correct_after": 0
}
```

This metric measures whether on-policy logits-level distillation helps fix examples that the cold-start student originally failed.

### 10.4 Reporting Compression Ratio

Compression ratio should be reported as a data-construction statistic, not as the main evaluation metric.

Recommended report format:
During cold-start data construction, the teacher first generates long CoTs and then compresses them into difficulty-aware rationales. We report average teacher-long-rationale length, compressed-rationale length, compression ratio, and token reduction overall and by difficulty bucket.

Do not present compression ratio as direct evidence of better reasoning performance.
The main performance evidence should come from final-answer accuracy and error recovery.


## 11. Answer Extraction

Implement robust answer extraction.

Preferred parsing order:
1. extract content after ### Answer;
2. if unavailable, use the last non-empty line;
3. if still unavailable, extract the last number in the response.

Normalize answers by:
1. stripping whitespace;
2. removing commas from numbers;
3. removing currency symbols;
4. removing trailing periods;
5. handling simple integer and decimal formats.
